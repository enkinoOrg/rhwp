# Markdown 기반 HWPX 템플릿 컴파일러 설계

## 1. 목적

공고문, 제안요청서, 평가표, 조사 자료와 샘플 HWPX를 입력받아 AI가 사업계획서의 목차와 본문 초안을 Markdown으로 작성하고, RHWP가 샘플 문서의 서식을 계승한 수정 가능한 HWPX를 생성한다.

이 기능의 목적은 최종 문서를 무인 완성하는 것이 아니다. 사용자가 자료 검토, 문서 작성 계획, 평가항목 대응, 상세 내용 검토와 페이지 예산 조정에 시간을 집중하도록 초안 작성과 반복 서식 작업을 줄이는 것이다. 최종 세부 편집은 RHWP에서 수행한다.

## 2. 범위와 전제

### 2.1 첫 버전의 전제

- 입력 템플릿은 HWPX로 제한한다.
- 표지, 회사 일반현황과 기존 소개자료는 원본 그대로 보존한다.
- 시스템이 본문 시작 후보 2~3개를 제시하고 사용자가 하나를 선택한다.
- 선택 위치 이전은 보존 구간, 이후는 AI 생성 본문으로 교체한다.
- AI는 기존 목차에 종속되지 않고 제목을 변경·추가·삭제할 수 있다.
- 생성 본문은 제목, 일반 문단, 핵심항목, 세부항목, 표와 이미지 자리표시자를 지원한다.
- 사용자는 생성 결과를 RHWP에서 최종 수정한다.

### 2.2 첫 버전에서 제외

- HWP 바이너리를 직접 템플릿으로 사용하는 기능
- 복잡한 셀 병합과 중첩 표
- 이미지 둘러싸기와 자유 배치
- 임의 도형과 차트 자동 생성
- 회사자료 자동 병합
- 기관 양식을 완전히 무인으로 판정하는 기능
- AI가 RHWP 스타일 ID나 편집 API를 직접 반복 조작하는 기능
- 템플릿 마켓, 다중 사용자 권한과 결제 기능

## 3. 설계 원칙

### 3.1 의미와 서식을 분리한다

Markdown은 제목, 본문, 목록, 표와 이미지라는 의미만 표현한다. `### = 1-1.`처럼 Markdown 문법과 HWP 출력 형식을 전역으로 고정하지 않는다. 번호, 글머리표, 글꼴, 크기, 들여쓰기와 여백은 샘플 HWPX에서 추출한 `StyleProfile`이 결정한다.

### 3.2 AI는 판단에만 사용한다

AI는 다음 작업을 담당한다.

- 공고문과 평가표 분석
- 조사 결과 요약
- 문서 청사진과 목차 설계
- 섹션별 본문 작성
- 평가항목 커버리지 판단
- 페이지 제한 초과 시 축약안 작성

다음 작업은 결정적 코드가 담당한다.

- Markdown 파싱과 정규화
- 번호 생성
- 스타일 매핑
- RHWP IR 구성
- 재조판과 HWPX 직렬화
- 라우팅, 재시도와 검증

같은 템플릿, Markdown과 옵션은 항상 같은 문서를 생성해야 한다.

### 3.3 실패를 숨기지 않는다

지원하지 않는 Markdown, 신뢰도가 낮은 스타일, 누락 이미지와 보존할 수 없는 HWPX 구조는 변환 보고서에 기록한다. 필수 근거가 없으면 내용을 만들어내지 않고 `[자료 필요]`로 표시한다.

## 4. 전체 아키텍처

```text
[AI 작성 계층]
공고문 + 평가표 + 조사자료
    → DocumentBlueprint
    → 일반 Markdown 초안

[문서 컴파일 계층]
Markdown
    → Markdown AST
    → 정규화된 Document AST
    → 번호 체계와 StyleProfile 적용
    → RHWP Document IR 패치

[렌더링·검증 계층]
RHWP Core 재조판
    → HWPX blob
    → 페이지·평가항목·레이아웃 검사
    → 사용자 수정 또는 AI 축약안 비교
```

### 4.1 컴포넌트 경계

#### TemplateAnalyzer

샘플 HWPX의 문단, 스타일, 번호, 표, 페이지와 구역 정보를 분석한다. 본문 시작 후보와 스타일 역할 후보를 신뢰도와 함께 반환한다.

#### TemplateProfileStore

사용자가 확정한 본문 시작 위치와 스타일 역할을 템플릿 지문에 연결한다. 첫 버전에서는 단일 프로파일 저장과 재사용만 지원한다.

#### BlueprintPlanner

공고문, 평가표, 조사자료와 페이지 제한으로 `DocumentBlueprint`를 만든다. 이 컴포넌트는 AI 호출 결과를 스키마로 검증한다.

#### MarkdownNormalizer

일반 Markdown을 지원 가능한 의미 구조로 제한한다. 미지원 표현은 단순화하거나 경고하며 조용히 버리지 않는다.

#### DocumentCompiler

Document AST에 번호와 `StyleProfile`을 적용하고 RHWP 문단, 표와 이미지 컨트롤을 만든다. AI를 호출하지 않는다.

#### TemplatePatcher

원본 IR의 보존 구간을 유지하고 선택된 경계 이후를 컴파일된 본문으로 교체한다.

#### DocumentValidator

재조판 결과의 페이지 수, 빈 절, 표 넘침, 이미지 누락과 평가항목 커버리지를 검사한다.

#### RevisionPlanner

페이지 제한 초과 시 보호할 내용과 축약할 내용을 판정하여 원문과 비교 가능한 축약안을 만든다. 사용자 승인 전에는 원문을 변경하지 않는다.

## 5. 템플릿 분석

### 5.1 분석 특징

문단마다 다음 특징을 수집한다.

- 글자: 글꼴, 크기, 굵기, 색상, 자간과 장평
- 문단: 좌우 여백, 첫 줄 들여쓰기, 정렬, 줄간격과 앞뒤 간격
- 머리: 번호·글머리표 종류, 수준과 번호 정의
- 배치: 페이지 위치, 쪽 나눔과 주변 공백
- 내용: 번호 패턴, 문장 길이와 종결 형태
- 주변: 앞뒤 문단, 표와 이미지와의 관계

### 5.2 스타일 역할

첫 버전은 다음 역할을 추출한다.

- `sectionHeading`
- `subsectionHeading`
- `body`
- `keyPoint`
- `detail`
- `highlight`
- `tableHeaderCell`
- `tableBodyCell`
- `imageCaption`

프로파일은 원본 `paraShapeId`, `charShapeId`, `numberingId`, 표 테두리와 셀 여백을 참조한다. 새로운 스타일은 필요한 경우에만 가장 가까운 원본 스타일을 복제하여 제한적으로 만든다.

### 5.3 본문 경계

시스템은 제목 계층, 목차, 쪽 나눔, 빈 문단과 표 밀도를 이용해 본문 시작 후보 2~3개를 만든다. 각 후보는 다음 정보를 제공한다.

- 후보 문단의 제목과 페이지
- 앞뒤 문단 미리보기
- 후보 선택 시 보존되는 마지막 요소
- 후보 선택 시 교체되는 첫 요소
- 신뢰도와 판정 근거

사용자가 선택한 경계는 템플릿 지문과 함께 저장한다. 같은 템플릿을 다시 사용하면 저장된 선택을 기본값으로 사용하되 변경할 수 있다.

### 5.4 원본 보존

경계 이전의 원본 IR은 재구성하지 않고 유지한다. 다음 리소스를 보존한다.

- 표지, 회사 소개, 기존 표와 이미지
- 용지, 여백, 구역과 단 설정
- 머리말, 꼬리말과 페이지 번호
- 글꼴, 글자 모양, 문단 모양과 번호 정의
- 테두리와 배경 리소스

HWPX 패키지 보존 계약은 엔트리 역할에 따라 구분한다.

- `version.xml`, `settings.xml`, `Preview/PrvText.txt`, `Preview/PrvImage.png`처럼 본문과 독립적인 passthrough 보조 엔트리는 존재하는 원본 엔트리의 바이트가 동일해야 한다.
- `Contents/content.hpf`는 본문 교체에 따라 manifest와 spine을 다시 생성해야 하므로 전체 바이트 동일성을 요구하지 않는다.
- `content.hpf`의 metadata(제목, 언어, 작성자, 주제, 설명, 생성·수정일 등)는 의미가 동일해야 한다.
- manifest의 리소스 `id`, `href`, MIME 유형, embedded 여부와 spine의 참조 순서는 의미가 동일해야 한다. `image/jpg`와 `image/jpeg` 같은 표준 MIME 별칭은 동등하게 취급한다.

## 6. DocumentBlueprint

AI가 장문을 쓰기 전에 다음 구조의 청사진을 만든다.

```yaml
title: AI 기반 공연 안전 운영 플랫폼 사업계획서
pageBudget: 20
sections:
  - id: planning-overview
    title: 기획 개요
    pageBudget: 4
    scoreWeight: 20
    requiredEvidence:
      - 공연 안전사고 통계
      - 기존 안전관리 체계의 한계
    children:
      - id: background
        title: 기획 배경 및 필요성
        targetPages: 2
        keyMessages:
          - 일반 산업 안전 기술 적용의 한계
          - 예방 중심 실시간 안전 플랫폼의 필요성
```

청사진은 공고문의 필수 항목, 평가표의 배점, 조사 결과, 샘플 문서 구성, 사용자 사업 정보와 전체 페이지 제한을 함께 반영한다. 사용자가 청사진을 승인한 후 섹션별 Markdown을 작성한다.

## 7. Markdown과 Document AST 계약

### 7.1 입력 정책

사용자와 AI는 일반 Markdown을 사용한다. 컴파일 전에 지원 가능한 Document AST로 정규화한다.

| Markdown | Document AST | 기본 의미 |
|---|---|---|
| `##` | `SectionHeading` | 대제목 |
| `###` | `SubsectionHeading` | 소제목 |
| 일반 문단 | `BodyParagraph` | 본문 |
| 1단계 목록 | `KeyPoint` | 핵심항목 |
| 2단계 목록 | `DetailPoint` | 세부설명 |
| 숫자 목록 | `ProcedureItem` | 순서가 있는 절차 |
| 인용문 | `Highlight` | 핵심성과·강조문 |
| 표 | `Table` | 표 |
| 이미지 | `FigurePlaceholder` | 이미지 또는 자리표시자 |
| 굵게 | `StrongRun` | 굵은 글자 구간 |
| 각주 | `Footnote` | 각주 |

실제 번호와 글머리표는 이 표가 아니라 `StyleProfile`이 결정한다.

### 7.2 목록 정규화

- 제목 직후의 1단계 목록은 `KeyPoint`로 해석한다.
- 핵심항목 아래의 2단계 목록은 `DetailPoint`로 해석한다.
- 짧은 병렬 명사 목록은 `Enumeration`으로 해석한다.
- 순서가 중요한 숫자 목록은 `ProcedureItem`으로 유지한다.
- 신뢰도가 낮은 해석은 기본 규칙으로 처리하되 변환 보고서에 표시한다.

### 7.3 표

첫 버전은 Markdown 표, 머리행 한 개, 텍스트와 굵은 글씨, 열 너비 자동 배분, 페이지 폭 초과 감지와 긴 행 분할을 지원한다. 템플릿에서 추출한 테두리, 셀 배경, 글꼴과 셀 여백을 적용한다.

셀 병합, 중첩 표와 셀 내부 이미지는 제외한다.

### 7.4 이미지

- `asset://id`가 실제 파일과 연결되면 템플릿 기본 비율로 삽입한다.
- 파일이 없고 설명이 있으면 공간을 확보한 자리표시자와 캡션을 삽입한다.
- 파일과 설명이 모두 없으면 문서에서 생략하고 경고한다.

첫 버전은 문단처럼 취급되는 인라인 이미지를 기본으로 한다. 정밀 위치, 둘러싸기와 크기는 사용자가 RHWP에서 조정한다.

## 8. 컴파일과 데이터 흐름

각 단계는 재현 가능한 산출물을 남긴다.

```text
입력 수집
  → template-profile.json
  → boundary-candidates.json
  → template-analysis-report.md
  → blueprint.yaml
  → evaluation-coverage.json
  → draft.md
  → asset-manifest.json
  → normalized-document.json
  → generated.hwpx
  → validation-report.json
```

원본 IR 수정 순서는 다음과 같다.

1. 원본 HWPX를 RHWP Document IR로 로드한다.
2. 사용자가 확정한 경계 이전의 요소를 유지한다.
3. 경계 이후의 기존 본문을 제거한다.
4. Document AST에서 생성한 문단, 표와 이미지 컨트롤을 삽입한다.
5. 스타일과 번호 리소스 참조를 검증한다.
6. 문단 재구성과 페이지네이션을 실행한다.
7. HWPX로 직렬화하고 다시 로드한다.
8. 재로드한 결과로 품질 검사를 실행한다.

현재 RHWP의 시험문제용 `build-from-ingest` 경로를 범용 문서 입력으로 확장하지 않는다. Document AST와 템플릿 패치 계층을 별도 경계로 두고 기존 RHWP Core의 IR, 재조판과 직렬화를 재사용한다.

## 9. 품질 검사와 페이지 최적화

### 9.1 검사 항목

- 전체 및 섹션별 페이지 예산
- 공고문의 필수 작성항목
- 평가표 항목별 대응 문단
- 고배점 항목의 분량 부족
- 제목만 있고 내용이 없는 절
- 동일 내용 반복
- 조사 근거가 없는 정량 수치
- 표의 페이지 폭 초과와 비정상 행 분할
- 이미지 자산 누락
- 빈 페이지와 마지막 페이지의 과도한 공백

### 9.2 페이지 제한

페이지 수는 글자 수로 추정하지 않고 RHWP Core의 실제 페이지네이션 결과를 사용한다.

보호 우선순위는 다음과 같다.

1. 공고문 필수 요구사항
2. 높은 배점의 평가항목
3. 정량 목표와 근거
4. 차별성과 수행 전략

축약 우선순위는 다음과 같다.

1. 반복된 배경 설명
2. 낮은 배점 항목의 장문 사례
3. 표와 본문에 중복된 내용
4. 의미가 겹치는 세부 목록

초과 시 원문과 축약본, 변경 이유, 보호된 평가항목과 예상 페이지 감소량을 비교한다. 사용자가 승인한 축약만 Markdown에 적용하고 다시 컴파일한다.

## 10. 오류 처리

- 신뢰도가 낮은 스타일은 후보를 제시한다.
- 미지원 Markdown은 정규화 결과와 경고를 남긴다.
- 지원하지 않는 HWPX 구조는 보존 가능 여부를 표시한다.
- 필수 자료가 없으면 `[자료 필요]`를 사용한다.
- 페이지 제한을 맞출 수 없으면 내용을 임의 삭제하지 않는다.
- 실패 단계의 입력과 보고서를 보존하여 재실행할 수 있게 한다.
- 생성 HWPX 재로드에 실패하면 결과를 성공으로 반환하지 않는다.

## 11. 구현 단계

### Phase 1: 기술 검증

실제 HWPX 한 개에서 스타일 프로파일을 추출하고, 고정 Markdown을 삽입하여 보존 구간과 스타일 계승을 검증한다.

### Phase 2: 범용 컴파일러

Markdown AST, Document AST, 제목·목록·표·이미지 변환과 변환 보고서를 구현한다.

### Phase 3: 문서 기획 연동

공고문, 평가표와 조사자료에서 Blueprint와 Markdown 초안을 만들고 평가항목 커버리지를 검사한다.

### Phase 4: 페이지 최적화

실제 재조판 결과에서 초과 구간을 분석하고 원문과 축약안을 비교하여 사용자가 선택하게 한다.

각 Phase는 독립적으로 검증하고 다음 단계로 넘어간다.

## 13. 테스트와 성공 기준

### 13.1 구조 보존

- 선택 경계 앞의 문단, 표와 이미지가 바뀌지 않는다.
- 용지, 여백, 머리말과 꼬리말이 유지된다.
- 경계 이전 문단의 텍스트·스타일 참조와 controls 전체의 안정적 심층 표현이 유지된다.
- `doc_info`와 BinData 등 전역 리소스의 의미가 유지된다.
- passthrough 보조 엔트리는 바이트 동일성을, `content.hpf`는 metadata·manifest 리소스 참조·spine의 의미 동일성을 검증한다.

### 13.2 스타일 계승

- 새 제목이 추출한 제목 스타일을 사용한다.
- 본문과 목록의 들여쓰기와 줄간격이 원본과 일치한다.
- 불필요한 새 스타일을 만들지 않는다.

### 13.3 내용 변환

- 지원하는 Markdown 제목, 본문과 2단계 목록을 손실 없이 변환한다.
- 표 셀 텍스트와 이미지 캡션을 유지한다.
- 미지원 문법을 조용히 누락하지 않는다.

### 13.4 페이지네이션과 회귀

- 생성 HWPX를 RHWP가 다시 로드할 수 있다.
- 저장과 재로드 후 Document AST의 의미 내용이 유지된다.
- 재로드 전후 페이지 수가 동일하다.
- 빈 페이지와 표 폭 초과를 탐지한다.
- 페이지 제한 초과 시 원문을 임의 삭제하지 않는다.
- 기존 RHWP의 일반 HWPX 편집과 저장 동작이 깨지지 않는다.

### 13.5 MVP 성공 기준

- 서로 다른 실제 HWPX 템플릿 3종 이상에서 동작한다.
- 올바른 본문 시작 위치가 상위 3개 후보 안에 포함된다.
- 원본 보존 구간의 구조적 변경이 0건이다.
- 제목, 본문과 목록의 스타일 역할 매핑 정확도가 90% 이상이다.
- 지원 문법의 무음 손실이 0건이다.
- 생성 HWPX의 RHWP 재로드 성공률이 100%다.
- 사용자가 글꼴, 들여쓰기와 번호 체계를 전면 수정하지 않아도 된다.
- 사용자가 내용 검토와 제한적인 레이아웃 조정에 집중할 수 있다.
